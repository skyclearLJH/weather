# 예상 강수량 방송 그래픽 자동화 — 프로젝트 인계 문서

이 문서 하나만 읽고도 작업을 이어받을 수 있도록 정리한 인계서다.
**표출 규칙의 정본은 [`rain-graphic-spec.md`](./rain-graphic-spec.md)** 이고, 이 문서는
그 규칙이 나온 배경 + 데이터 소스 + 구현 구조 + 아직 안 된 일 + 작업 방식을 담는다.

---

## 0. 한 줄 요약

> 기상청 **단기예보 통보문(텍스트)** 을 읽어 **방송용 "예상 강수량" 지도 그래픽(16:9)** 을
> 자동 생성하는 것이 목표다. 현재 **렌더러는 완성**, **통보문 파서는 미구현**이라
> 사람이 통보문을 해석해 데이터를 손으로 입력하는 반자동 상태다.

읽는 순서 추천: 이 문서 §1~§3 → `rain-graphic-spec.md` 전체 → 이 문서 §5~§8.

## 1. 배경과 목표

- **누가 쓰나**: KBS 재난미디어센터. 기상 방송 그래픽 제작에 쓴다.
- **기존 방식**: 디자이너가 통보문을 보고 손으로 지도 그래픽을 만든다. 비 오는 날마다 반복 작업.
- **목표**: 통보문 → 초안 그래픽 자동 생성 → 사람이 라벨 위치만 미세 조정하는 **반자동(A안)**.
  100% 자동 생성이 아니라 "사람이 손볼 수 있는 좋은 초안"이 목표라는 점이 중요하다.
- **왜 반자동인가**: 라벨 배치는 미적 판단이 많이 들어가고, 통보문 표현이 매우 다양해
  완전 자동화 시 오표현 위험이 크다. 사용자도 이 방향에 동의했다.

## 2. 현재 상태 (무엇이 되고 무엇이 안 되나)

| 구성요소 | 상태 | 위치 |
|---|---|---|

| 세부구역→시군 매핑 | ⚠️ 부분 (필요한 것만 그때그때 추가) | `rainGraphicData.js` 상단 상수들 |

```

```

**이번 프로젝트가 채워야 할 칸은 "통보문 → 데이터" 자동화**다.

## 3. 데이터 소스 — 기상청 통보문

### 엔드포인트 (검증된 파라미터)

프로젝트는 Cloudflare Pages Function 프록시(`/api/kma/`)를 통해 기상청 API허브를 호출한다.
인증키는 서버에서 주입되므로 클라이언트는 키를 몰라도 된다.

```
GET /api/kma/api/typ01/url/fct_afs_ds.php
      ?tmfc1=YYYYMMDDHH   # 발표시각 범위 시작 (KST, 시 단위)
      &tmfc2=YYYYMMDDHH   # 발표시각 범위 끝
      &stn=108            # 발표관서 (108 = 전국/서울)
      &disp=0
      &help=1
```

실제 성공 예시:

```
https://weather-ljh.pages.dev/api/kma/api/typ01/url/fct_afs_ds.php
  ?tmfc1=2026071617&tmfc2=2026071618&stn=108&disp=0&help=1
```

**주의점 (실제로 겪은 함정)**

- 응답 인코딩은 **EUC-KR**이다. UTF-8로 읽으면 깨진다. (`iconv -f euc-kr`)
- `tmfc=`(단수) 파라미터로는 빈 응답(`#START7777` / `#7777END`만)이 온다.
  반드시 **`tmfc1`/`tmfc2` 범위**로 조회해야 한다.
- 통보문은 하루 여러 번(05/11/17시 등) 발표된다. 최신 발표를 쓰려면 범위를 최근으로 잡고
  가장 늦은 `TM_FC`를 고른다.
- 기존 구현 참고: `src/api/weatherApi.js`의 `fetchWeatherDocFromApiHub` (약 1883행~).
  응답 파싱은 같은 파일의 `parseKmaReport(rawText, stn, 7)`.

### 응답 형식

```
#START7777
#-------------------------------------------------
#  1. STN_ID : 발표관서 / 2. TM_FC : 발표시각(KST) ...
#-------------------------------------------------
$0#108#202607161700#202607161700#2#조경모#
$1#
□ (종합) 내일까지 남부지방 중심 비, 모레부터 전국 강하고 많은 비
 ○ (오늘, 16일) 전국 대체로 흐리고, ...

* 예상 강수량(16~17일)
 - (수도권, 16일) 경기남서부: 5mm 안팎
 - (충청권) 대전.세종.충남, 충북: 20~60mm
 - (전라권) 전북, 광주.전남: 30~80mm
 - (경상권) 부산.울산.경남: 30~80mm/ 대구.경북: 20~60mm
 - (제주도, 17일) 제주도: 5~10mm

* 예상 강수량(18일)
 - (수도권, 17일부터) 서울.인천.경기도: 50~150mm(많은 곳 경기남부 200mm 이상)/ 서해5도: 20~60mm
 - (강원도) 강원도: 30~80mm(많은 곳 강원중.남부내륙.산지 150mm 이상)
 ...
=
#7777END
```

우리가 쓰는 건 **`* 예상 강수량(...)` 블록들**뿐이다. `□ (종합)` / `○` 본문은 참고용.

## 4. 표출 규칙 — 정본은 별도 문서

**[`rain-graphic-spec.md`](./rain-graphic-spec.md) 를 반드시 정독할 것.** 통보문 10세트와
완성 그래픽을 대조해 뽑아낸 규칙이 전부 들어 있고, 사용자가 하나하나 확정한 내용이다.

여기서는 **가장 자주 틀리는 핵심만** 요약한다:

### 4.1 제작 여부
- `* 소나기에 의한 예상 강수량` 블록은 **전부 제외**
- 통보문 어딘가에 (많은 곳 포함) **100mm 이상**이 있으면 그 통보문은 제작 대상
- 제작 대상이면 **날짜 블록마다 1장씩**
- 단, 블록 최대값 **20mm 미만**이면 생략 / **권역 1개뿐이고 100mm 미만**이면 생략

### 4.2 라벨 (틀리기 쉬움)
- 기본 형식은 `최소~최대`. 많은 곳 값은 **본체 라벨에 합치지 않고** 해당 세부구역에 `N↑`로 분리
  - 예: 수도권 본체 `50~150`, 경기남부만 `200↑`
- **같은 값 범위는 지도 전체에서 라벨 1번만** 표시
- `Nmm 내외/미만/안팎` → 전부 `N안팎`
- 기간 주석은 **대상일을 넘겨 연장될 때만** `(~내일 아침)` / `(내일부터)` 식으로, 수치보다 작게

### 4.3 색
- **상대 배색**: 색 서열(연하늘→하늘→파랑→남색→보라→자주→검보라)은 고정,
  그날 등장한 값 단계를 낮은 것부터 순서대로 배치. 절대값-색 고정 매핑이 아니다.
- 강수 예보 없는 권역은 **회색**

### 4.4 절대 하지 말 것 ⚠️
디자이너 완성본에는 **임의 단순화**가 섞여 있다. 이걸 규칙으로 오해하고 구현하면 안 된다:
- 서로 다른 범위를 합집합으로 병합 (`20~60` + `30~80` → `20~80`) ❌
- 라벨 축약 (`50~120(150↑)` → `50~150↑`) ❌
- 세트1 제주 라벨 `10~60(10↑)`는 **디자이너 오타** (정답 `10~60(100↑)`)

`rain-graphic-spec.md`의 "디자이너 임의 단순화" 절에 사례가 정리돼 있다.

## 5. 구현 구조

### 5.1 파일

| 파일 | 역할 |
|---|---|
| `src/data/rainGraphicData.js` | 그래픽 정의(구역 선택자 + 색 + 라벨). **현재 수작업 입력** |
| `src/components/RainGraphicView.jsx` | 렌더러. GeoJSON을 SVG path로 투영해 그림 |
| `src/components/RainGraphicView.css` | 레이아웃(배경/타이틀/라벨 박스) |
| `public/data/map/kr-{sido,sgg,emd}-20260701.geojson` | 경계 데이터(통계청 SGIS, CC BY 4.0) |
| `public/generated/rainfall/` | 완성본 + 배경 이미지 |

### 5.2 진입 방법

```
https://weather-ljh.pages.dev/?rainGraphic=20260719-19-20
로컬:  http://localhost:5173/?rainGraphic=20260719-19-20
```

`src/main.jsx`에서 `?rainGraphic=<ID>` 쿼리로 게이트한다(일반 UI엔 노출 안 함).
ID는 `RAIN_GRAPHICS`의 키.

### 5.3 데이터 스키마

```js
RAIN_GRAPHICS['20260719-19-20'] = {
  title: '예상 강수량',
  period: '(~내일, mm)',        // 좌측 타이틀 아래 기간 표기
  issuedAt: '2026-07-19',
  layers: [                     // 그리는 순서대로 (뒤가 위)
    {
      id: '100-200',
      color: '#63348e',
      selector: province('강원특별자치도'),
      emphasis: true,           // '많은 곳' 구역 표시용
    },
  ],
  labels: [
    { text: '300↑', note: '(19일 밤부터)', lon: 127.25, lat: 37.72,
      color: '#54116e', darkText: false },
  ],
};
```

### 5.4 구역 선택자 DSL

`rainGraphicData.js` 상단에 정의된 3개 헬퍼로 구역을 조합한다:

```js
province('강원특별자치도', '충청북도')        // 시도 전체
cities('경기도', '수원시', '성남시', ...)      // 시군구 조합 (startsWith 매칭)
mergeSelectors(A, B, ...)                     // 합치기
{ emdCodes: ['32030370', ...] }               // 읍면동 단위 (산지 등 정밀 구역)
```

이미 만들어둔 세부구역 상수들(재사용할 것):

```
GYEONGGI_SOUTH / GYEONGGI_SOUTHWEST / GYEONGGI_NORTH
CHUNGNAM_NORTH / CHUNGBUK_CENTRAL_NORTH
GANGWON_CENTRAL_SOUTH_INLAND / GANGWON_NORTH_INLAND / GANGWON_EAST_COAST
GANGWON_CENTRAL_SOUTH_MOUNTAIN_EMD / GANGWON_NORTH_MOUNTAIN_EMD   ← 산지는 읍면동 단위
GYEONGBUK_CENTRAL_NORTH / GYEONGBUK_NORTH / GYEONGBUK_SOUTH / GYEONGBUK_MAINLAND
JEONBUK_WEST / JEONBUK_EAST / JEONBUK_NORTHWEST_COAST
GYEONGNAM_EXCEPT_SOUTH_COAST / ULLEUNG_AND_DOKDO / WEST_SEA_FIVE_ISLANDS
```

> 참고: `GYEONGBUK_MAINLAND`는 울릉군을 뺀 경북. 통보문에 `울릉도.독도`가
> 명시되지 않은 블록에서 울릉도를 회색으로 두기 위해 만들었다.

### 5.5 투영

`RainGraphicView.jsx`는 단순 등장방형(위경도 선형) 투영을 쓴다.

```js
MAP_BOUNDS = { minLon: 124.45, maxLon: 131.05, minLat: 32.95, maxLat: 38.85 }
longitudeScale = cos(36°)   // 경도 압축
```

`projectCoordinate([lon, lat]) → [x, y]` (1040×1000 뷰박스). 라벨도 같은 함수로 배치하므로
`labels[].lon/lat`만 주면 된다.

## 6. 아직 안 된 일 (= 이번 프로젝트의 실제 과제)

### 6.1 통보문 파서 ★ 최우선

`* 예상 강수량(...)` 블록을 구조화된 데이터로 바꾸는 것. **변형이 매우 많다.**
`rain-graphic-spec.md`의 "통보문 파싱 패턴" 절에 **실제로 관측된 변형이 전부 나열**돼 있다.
그걸 그대로 테스트 케이스로 만들 것을 강력히 권한다.

특히 까다로운 것들:
- 기간 한정이 권역 옆 / 구역 옆 / 권역 옆 일부 구역만 등 위치가 제각각
- `많은 곳`이 다중이고 값이 다름: `많은 곳 전북 120mm 이상, 광주.전남북부 100mm 이상`
- 괄호 제외 표현: `광주.전남(남서부 제외)`, `전남해안(북부서해안 제외)`
- 점(.)으로 3개 이상 연결된 복합 구역명: `부산.경남남해안.경남남서내륙`
- 시군 경계와 무관한 지형 명칭: `지리산부근` → 매핑 불가 시 본체 라벨에만 반영

목표 출력 스키마(제안):

```js
{
  blockDate: '16~17일',
  items: [
    { region: '수도권', dateScope: '16일',
      areas: ['경기남서부'], min: 5, max: 5, around: true },
    { region: '수도권', dateScope: '17일부터',
      areas: ['서울','인천','경기도'], min: 50, max: 150,
      heavy: [{ areas: ['경기남부'], value: 200 }] },
  ],
}
```

### 6.2 세부구역 → 시군 매핑 테이블

`강원중.남부내륙`, `충남북서부`, `경북남서내륙` 같은 예보구역명을 시군(또는 읍면동) 조합으로
바꾸는 표가 필요하다. 지금은 필요한 것만 그때그때 상수로 만들어 놨다(§5.4).

**기상청 공식 예보구역 정의를 기준으로 만들 것.** 산지처럼 시군보다 작은 단위는
읍면동 코드(`emdCodes`)를 쓴다 — `GANGWON_CENTRAL_SOUTH_MOUNTAIN_EMD`가 예시다.

### 6.3 상대 배색 자동 결정

그날 등장한 값 단계를 모아 정렬하고 고정 색 서열에 순서대로 매핑하는 로직.
현재는 색을 손으로 넣는다. 규칙은 §4.3 / 스펙 문서 참조.

### 6.4 라벨 자동 배치

`rain-graphic-spec.md`의 **"라벨 배치 우선순위"** 절(8개 항목)이 사실상 알고리즘 명세다.
비용 함수까지 정의돼 있다:

```
다른 값 구역 침범 > 다른 라벨과 겹침 > 대상 구역을 과도하게 가림 > 대상 구역 중심에서 멀어짐
```

작은 구역부터 배치하고, 후보는 `대상 구역 내부 → 같은 값 인접 구역 → 인접 바다` 순으로 탐색.

## 7. 작업 방식 / 사용자 선호 (실제로 지적받은 것들)

이 프로젝트를 진행하며 사용자가 명시적으로 정한 것들이다. 지키면 재작업이 준다.

1. **디자이너 완성본을 그대로 베끼지 말 것.** 완성본에는 임의 단순화와 오타가 섞여 있다.
   규칙(원칙형)과 재량(단순화)을 구분해야 한다. (§4.4)
2. **라벨은 항상 원칙 형식**을 쓴다. 축약형은 디자이너 재량이지 규칙이 아니다.
3. **작업 후 자동 커밋·푸시.** 묻지 말고 `main`에 푸시하면 Cloudflare Pages가 자동 배포한다.
   커밋 메시지는 영어 명령형 한 줄 제목.
4. **다른 AI(Codex)와 병행 작업**한다. 작업 시작 전 반드시 `git fetch`로 최신 확인.
   같은 로컬 폴더를 공유하므로 내 미커밋 변경이 이미 반영돼 있을 수 있다.
5. **결과를 눈으로 검증하고 보고할 것.** 스크린샷 없이 "됐습니다"라고 하지 말 것.
   (실제로 링 아티팩트를 놓치고 넘어간 적이 있어 지적받았다.)
6. 새 규칙을 확정하면 **`rain-graphic-spec.md`에 즉시 반영**한다. 그게 정본이다.

## 9. 열린 질문 (사용자 확인이 필요한 것)

- **같은 값 인접 '많은 곳' 구역의 라벨 병합** — 스펙에 "채택 여부 확인 권장"으로 남아 있다.
  이후 피드백에서 "200↑과 150↑은 인접이어도 따로" 로 정리됐으나, **같은 값끼리** 병합은
  아직 명시적 확정이 없다. (`rain-graphic-spec.md` 라벨 규칙 절 참조)
- 세부구역 매핑을 어느 정밀도까지 갈지 (시군 vs 읍면동). 산지·해안은 읍면동이 필요하다.
- 완전 자동 배치까지 갈지, 초안 후 사람이 라벨을 끌어 옮기는 UI를 둘지.
